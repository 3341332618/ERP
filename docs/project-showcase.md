# ERP 管理平台：项目展示与答辩说明

本文用于课程验收、课堂展示和项目答辩。运行、配置、API 和部署细节请查看根目录 [README](../README.md)。

## 一句话介绍

这是一个以“采购—库存—销售—结算”为核心闭环的前后端分离 ERP 系统，并加入了可发布真实缺陷、隔离学员数据和自动累计成绩的软件测试竞赛模块。

## 项目价值

传统的管理系统演示常停留在增删改查。本项目重点展示业务数据之间的联动：

- 业务单据必须经过状态流转和仓库审核。
- 审核通过才会改变实际库存。
- 出入库会自动生成对应的收入或支出结算。
- 退货必须关联原始单据并校验剩余可退数量。
- 调拨会同时改变两个仓库的库存，并通知对应仓库人员。
- 测试学员在独立工作区操作，不会污染其他学员或正式演示数据。

## 展示重点

### 1. 完整 ERP 闭环

```text
维护基础资料
  → 创建采购/销售单据
  → 提交仓库审核
  → 更新实际库存
  → 自动生成收支结算
  → 查看关联业务详情
```

### 2. 角色驱动的工作台

系统登录后由后端按角色返回菜单：

| 角色 | 展示重点 |
| --- | --- |
| 系统管理员 | 全部 ERP 模块、缺陷发布、学员和报告评分 |
| 采购/销售人员 | 创建并维护自己的业务单据 |
| 仓库主管 | 查看库存、入库审核和出库审核列表 |
| 仓库专员 | 接收仓库消息，执行入库或出库审核 |
| 结算主管 | 查看自动生成的收入和支出记录 |
| 终极管理员 | 测试文件评阅、操作轨迹和评分历史 |
| 测试学员 | 独立 ERP 工作区、测试任务和材料提交 |

### 3. 可验证的业务约束

推荐现场重点演示三条：

1. 销售出库数量超过可用库存时，系统阻止审核通过。
2. 退货单没有关联已审核原单，或数量超过剩余可退量时，系统拒绝操作。
3. 商品存在库存或未完结单据时，系统阻止停用。

### 4. 软件测试竞赛闭环

```text
管理员发布缺陷
  → 学员领取测试任务
  → 学员复现缺陷并提交报告/文件
  → 管理员或终极管理员评分
  → 生成排行榜、评分历史和操作轨迹
```

预置缺陷由 `backend/src/main/resources/bug-training-definitions.json` 加载。缺陷开关会改变实际业务行为，不只是页面上的静态题目。

## 技术亮点

- Spring Security + JWT 实现无状态认证。
- BCrypt 保存用户密码，前端自动携带 Bearer Token。
- 统一 API 响应格式和全局业务异常处理。
- Vue Router 路由守卫阻止未登录访问。
- Pinia 维护当前用户、Token 和角色菜单。
- 业务规则集中在后端执行，前端不承担最终可信校验。
- 自动化测试覆盖库存、退货、调拨、结算和竞赛关键链路。
- 学员工作区通过数据所有者字段实现逻辑隔离。

访问 <http://127.0.0.1:5173>，先使用 `admin / 123456` 登录。

后端重启会恢复内置账号和基础资料，适合在展示前重置环境。竞赛上传文件不会自动删除，但重启后旧文件不会出现在页面记录中。

## 推荐演示流程

下面是一套约 8 至 10 分钟的完整流程。如果时间不足，可只演示前六步。

### 第 1 步：登录和角色菜单（约 40 秒）

1. 使用 `admin / 123456` 登录。
2. 展示首页、左侧业务菜单、右上角用户角色和系统消息入口。
3. 说明菜单由后端根据角色动态返回，而不是只靠前端隐藏。

讲解句：

> 系统采用 JWT 无状态认证。用户登录后，后端同时返回用户信息和角色菜单，后续请求由前端统一携带 Token。

### 第 2 步：基础资料（约 50 秒）

1. 进入商品管理，展示品牌、分类、单位和价格信息。
2. 进入仓库、客户和供应商页面。
3. 展示搜索、状态切换或商品导入入口。

讲解句：

> 基础资料是采购、销售和库存单据的来源；后端会校验引用对象及其状态，避免无效数据进入业务流程。

### 第 3 步：采购入库（约 90 秒）

1. 切换到 `purchase_staff / 123456`。
2. 创建采购入库单，选择仓库、供应商和商品，填写数量后保存。
3. 提交单据，展示状态从“待提交”变为“待审核”。

讲解句：

> 保存单据不会立即增加库存，只有仓库审核通过后才改变实际库存。

### 第 4 步：仓库审核和库存变化（约 90 秒）

1. 切换到 `warehouse_staff / 123456`。
2. 打开消息入口，展示待审核通知。
3. 进入“入库审核”，对刚才的采购单执行审核通过。
4. 打开库存分布，确认商品实际库存增加。

讲解句：

> 仓库人员只能处理与仓库匹配的单据。审核动作同时触发库存更新和后续结算，保证业务状态一致。

### 第 5 步：自动结算（约 50 秒）

1. 切换到 `settlement_manager / 123456`。
2. 进入支出结算，找到采购入库自动生成的记录。
3. 打开详情，展示关联单据号和原业务数据。

讲解句：

> 采购入库对应支出，销售出库对应收入；结算记录由审核动作自动生成，减少人工重复录入。

### 第 6 步：退货或库存调拨（约 80 秒）

任选一个流程：

- 创建采购退货单，关联刚才已审核的采购入库单，展示可退数量校验。
- 创建库存调拨单，展示调出仓和调入仓必须不同，以及审核后的双向库存变化。

讲解句：

> 退货不是独立的随意出入库，而是基于原单控制数量；调拨则在一次审核中原子地完成两个仓库的库存变更。

### 第 7 步：测试竞赛（约 2 分钟）

1. 使用 `admin / 123456` 在缺陷库发布一个测试任务。
2. 使用 `student01 / 123456` 查看任务并提交缺陷报告。
3. 切回 `admin` 对报告评分，展示排行榜变化。
4. 如需展示文件流程：学员上传 PDF/Word/Excel，使用 `superadmin / 123456` 完成文件评阅，并查看操作轨迹和评分历史。

讲解句：

> 竞赛模块复用 ERP 作为真实测试对象。学员数据彼此隔离，评分通过后自动进入排行榜，终极管理员还能追踪操作和历史成绩。

## 三分钟精简版

1. `admin` 登录并展示模块全景。
2. 打开一张采购入库单说明状态流转。
3. 切到仓库审核并说明库存只在通过后变化。
4. 打开支出结算说明自动联动。
5. 用竞赛排行榜收尾，说明项目同时支持软件测试实训。

## 答辩陈述参考

### 开场（约 30 秒）

> 我们实现的是一个前后端分离的 ERP 管理平台。系统覆盖基础资料、采购、销售、库存和结算，不只是页面增删改查，还实现了单据状态、仓库审核、库存扣减、退货关联和收支结算等业务联动。在此基础上，我们又扩展了软件测试竞赛模块，让学员可以在隔离环境中测试真实业务缺陷并提交报告，由管理员评分和生成排行榜。

### 架构说明（约 40 秒）

> 后端使用 Spring Boot 3，Spring Security 和 JWT 负责登录鉴权，核心业务规则集中在服务端。前端使用 Vue 3、TypeScript、Pinia、Vue Router 和 Element Plus。开发时由 Vite 把 `/api` 代理到后端。当前演示版本使用内存数据，降低启动门槛；生产化时可以把存储层替换为数据库实现。

### 收尾（约 20 秒）

> 项目的重点是把多个业务模块连接成可验证闭环。我们通过 18 个后端测试覆盖采购、销售、库存、退货、调拨、结算和竞赛流程，前端也能完成生产构建。后续重点会放在数据库持久化、权限粒度、接口 DTO 和前端分包优化上。

## 常见答辩问题

### 为什么没有使用数据库？

当前版本定位为可快速运行的课程和测试演示环境，因此使用内存存储并预置数据，避免额外安装数据库。业务边界已集中在 `ErpStore`，后续可以拆分 Repository 并接入 MySQL 或 PostgreSQL。

### 如何保证库存不会被随意修改？

库存变更只发生在后端审核通过逻辑中。出库前会检查库存是否充足；可用库存还会预扣未完结的出库类单据，减少重复占用。

### 如何处理退货数量超过原单的问题？

采购退货和销售退货必须关联相应的已审核原单。后端会累计其他未拒绝退货单的数量，并校验本次退货是否超过原单剩余数量。

### 权限只是在前端隐藏菜单吗？

不是。前端菜单用于改善操作体验，所有 API 都要求 JWT；业务层还会检查管理员、终极管理员、学员、仓库人员、单据所有者和工作区归属。

### 学员数据如何隔离？

学员创建的基础资料、单据、库存和结算都带有工作区所有者标识。查询和业务操作会按当前学员过滤；普通 ERP 演示数据与学员数据也相互隔离。

### 测试竞赛的缺陷是真实的吗？

缺陷定义从 JSON 资源加载。管理员发布后，部分缺陷开关会实际改变后端业务行为，自动化测试也验证了开关前后的差异。

### 项目如何测试？

后端使用 Spring Boot 集成测试验证核心闭环和竞赛流程；前端使用 TypeScript 类型检查、Vite 生产构建和 Vitest 测试脚本。当前已核验后端 18 个测试通过，前端生产构建通过。

### 如果要投入生产，还需要做什么？

优先补充数据库持久化和迁移、细粒度授权、刷新 Token、审计日志、对象存储、OpenAPI、监控告警、容器化部署和备份恢复方案。

## 现场故障预案

- 登录失败：确认使用内置账号，密码为 `123456`；重启后端可恢复初始密码。
- 页面提示请求失败：确认后端 `8080` 和前端 `5173` 均已启动。
- 演示数据混乱：重启后端恢复初始内存数据，再重新走一遍最短流程。
- 端口冲突：提前用 `Get-NetTCPConnection -LocalPort 8080,5173` 检查。
- 网络不可用：项目启动后不依赖外部服务；提前完成 Maven 和 npm 依赖下载即可。
- 时间不足：切换到“三分钟精简版”，重点保留审核、库存和结算联动。
