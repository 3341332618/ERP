# 账号密码补齐与重置设计

## 目标

补齐历史数据库中的用户密码摘要，给终极管理员提供将学员及其 ERP 子账号统一重置为 `123456` 的入口，并修复禁用账号仍可使用的问题。现有 ERP 业务模块和内存/数据库双写架构不在本次重构范围内。

## 数据库设计

用户密码继续保存在 `sys_user.password_hash`，类型为 `VARCHAR(100)`，内容为 BCrypt 摘要，不新增明文密码列。

新增 Flyway V5 迁移：

1. 兼容旧库中缺少 `password_hash` 字段的情况并补齐字段。
2. 只把 `NULL` 或空字符串摘要回填为固定密码 `123456` 的 BCrypt 摘要。
3. 不覆盖已有摘要，避免重置用户已经修改的密码。
4. 回填后将字段恢复为 `NOT NULL`。

## 后端设计

新增 `POST /api/competition/students/{studentId}/reset-password`，仅 `SUPER_ADMIN` 可以调用。服务先确认目标是学员主账号，然后查找该主账号及 `workspaceOwnerId` 指向它的全部 ERP 子账号，为每个账号生成 `123456` 的 BCrypt 摘要并持久化。

响应返回重置数量和固定结果提示，不返回数据库摘要。不存在的学员、非学员目标或无权限调用统一返回现有业务错误结构。

登录时检查账号状态，只有 `ENABLED` 可以获得 Token。JWT 过滤器每次恢复用户身份时再次检查状态，使禁用账号持有的旧 Token 立即失效。

登录、修改密码和新增学员请求补齐非空与 6 至 72 位长度校验；修改密码继续要求两次新密码一致。新增学员密码为空时仍使用 `123456`。

## 前端设计

学员管理表新增“重置密码”操作。确认弹窗明确说明学员主账号和全部 ERP 子账号都会恢复为 `123456`。成功后显示实际重置数量并刷新学员列表；失败时保留页面状态并显示后端错误。

密码不会在列表或普通用户接口长期展示。管理员只从确认文案和成功提示中得知重置后的固定密码。

## 测试设计

- V5 能补齐缺失或空密码摘要，且不覆盖已有摘要。
- 重置学员密码会覆盖主账号及其全部 ERP 子账号。
- 其他学员和平台账号不受影响。
- 重置后所有关联账号可使用 `123456` 登录，旧密码失效。
- `DISABLED` 账号无法登录，已有 Token 也不能继续访问。
- 请求参数校验、前端 API 调用、确认文案和操作入口有回归测试。
