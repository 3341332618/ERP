# 业务关联下拉与详情查看设计

## 目标

本轮只优化 ERP 业务关联输入与详情查看，不调整竞赛页面、侧栏、主题或全站响应式布局。

目标包括：

- 将只能引用现有业务数据的自由输入框改为可搜索下拉。
- 让前端候选项与后端业务约束保持一致。
- 为基础资料补齐只读详情，并允许从退货单继续查看关联原单。
- 保留现有数据模型和 API 风格，避免大范围数据库关联重构。

## 非目标

- 不把商品分类、品牌、单位改为新的 ID 请求字段或新增数据库迁移。
- 不重构全部业务 Controller 的 `Map` 请求为 DTO。
- 不修改测试竞赛页面、导航布局、视觉主题和移动端整体结构。
- 不向学员暴露预置缺陷或其他跨工作区数据。

## 实现方案

采用按现有接口补强的全栈方案：主数据和库存选项复用现有列表接口；退货原单增加专用后端接口；所有提交继续由后端执行最终校验。

### 主数据关联

商品新增和修改中的分类、品牌、单位改为可搜索下拉：

| 表单字段 | 数据来源 | 提交值 |
| --- | --- | --- |
| 商品分类 | `GET /api/masterdata/category?status=ENABLED` | `categoryName` |
| 商品品牌 | `GET /api/masterdata/brand?status=ENABLED` | `brandName` |
| 商品单位 | `GET /api/masterdata/unit?status=ENABLED` | `unitName` |

候选项按名称去重；学员工作区优先显示本地记录。选择系统公共记录时，后端可生成该学员工作区的本地引用副本，但不得再根据任意文本创建不存在的分类、品牌或单位。

### 普通业务单据

| 单据 | 仓库 | 往来单位 | 商品 |
| --- | --- | --- | --- |
| 采购入库 | 启用仓库 | 启用供应商 | 启用商品 |
| 销售出库 | 启用仓库 | 启用客户 | 所选仓库中可用库存大于 0 的启用商品 |
| 库存调拨 | 有可用库存的启用调出仓库 | 调入仓库排除调出仓库 | 调出仓库中可用库存大于 0 的启用商品 |

下拉显示“编号 - 名称”，值使用记录 ID。更换仓库后必须清空已经失效的商品选择。

### 退货业务

新增接口：

```http
GET /api/documents/{type}/return-options?editingId={optionalId}
```

`type` 只允许 `purchase-return` 或 `sales-return`。接口只返回当前登录账号可用的原单：

- 与当前工作区一致。
- 采购退货只返回采购入库，销售退货只返回销售出库。
- 原单状态必须为 `APPROVED`。
- 原单创建人必须是当前业务账号。
- 至少存在一个剩余可退数量大于 0 的商品。

响应结构：

```json
[
  {
    "documentId": 1,
    "documentNo": "CGRK202607110001",
    "warehouseId": 10,
    "warehouseCode": "CK001",
    "warehouseName": "华东仓库",
    "partnerId": 20,
    "partnerCode": "GYS001",
    "partnerName": "示例供应商",
    "items": [
      {
        "productId": 30,
        "productCode": "SP001",
        "productName": "示例商品",
        "unitName": "件",
        "price": 100.00,
        "originalQuantity": 10.00,
        "returnedQuantity": 2.00,
        "remainingQuantity": 8.00
      }
    ]
  }
]
```

`editingId` 仅用于编辑已有退货单时排除该单自身数量，避免 unchanged save 被重复扣减。后端必须验证该 ID 属于当前用户、当前退货类型和当前工作区。

退货表单先选择原单，再只读展示仓库和客户/供应商。商品只能从原单剩余可退项中选择，价格自动使用原单价格，数量上限为 `remainingQuantity`。

## 后端约束

前端下拉只改善交互，后端仍是最终规则来源：

- 普通新增或变更关联只能使用当前工作区可见且状态为 `ENABLED` 的仓库、客户、供应商和商品。
- 商品分类、品牌、单位必须匹配已存在且启用的主数据；不再自动创建任意名称。
- 学员选择可见的系统公共主数据时，应先解析为该学员工作区的本地副本，再用于真实数据库外键。
- 调拨的调出、调入仓库不能相同。
- 销售出库、采购退货和调拨数量不能超过当前可用库存。
- 退货原单类型、状态、创建人、工作区、商品和剩余数量必须在保存时重新校验，防止选项加载后的并发变化。
- 退货商品价格必须由后端使用原单对应商品价格，忽略客户端自行提交的退货价格。
- `GET /api/documents/{type}/{id}` 必须校验路径 `type` 与真实单据类型一致。

编辑过程中关联资料若已停用，前端显示当前值并标记“已停用”，但保存前必须重新选择有效数据。

## 前端交互

### 单据表单

- 仓库、往来单位、商品和退货原单使用 `el-select`，启用 `filterable`，不允许创建任意选项。
- 数量和价格使用 `el-input-number`。普通采购/销售选择商品后自动带出建议价格，价格仍可修改。
- 调拨不显示价格输入；备注继续按现有后端规则校验。
- 下拉没有候选项时显示“请先维护可用的对应基础资料”。
- 上游选项变化时清空失效的下游字段。
- 表单使用 `ref` 调用 `validate()`，下拉字段校验触发器使用 `change`。
- 保存失败时保留弹窗和用户输入，刷新候选数据并显示后端错误。

### 详情查看

- 商品、仓库、客户、供应商、品牌、分类、单位列表操作列增加“查看详情”。
- 详情使用只读弹窗，按资料类型展示编号、名称、状态、业务字段、图片和创建/更新时间。
- 退货详情中的关联单据号增加“查看原单”，通过已有单据详情接口打开来源单据详情。
- 现有单据、审核、库存和结算详情能力保持不变。

## 错误处理

后端业务错误应使用可操作的中文消息，例如：

- `仓库已停用，请重新选择。`
- `商品不属于所选仓库或当前无可用库存。`
- `关联原单已无可退商品，请刷新后重试。`
- `关联单据类型不匹配。`

选项过期时不关闭表单。前端显示错误后刷新相应候选项，并清理已失效的选择。

## 测试

### 后端

- 只允许启用且当前工作区可见的关联资料。
- 任意分类、品牌、单位文本不会自动创建主数据。
- 系统公共主数据在学员工作区正确解析为本地引用。
- 可退原单只包含本人、正确类型和已审核记录。
- 剩余可退数量扣除其他未驳回退货，编辑时排除当前退货单。
- 保存时拒绝过期、跨工作区、错误类型或超量退货选择。
- 退货保存使用原单价格，客户端提交其他价格不能覆盖。
- 单据详情路径类型与真实类型不一致时拒绝访问。

### 前端

- 主数据和单据关联字段使用固定候选下拉，不包含 `allow-create`。
- 仓库变化会清空商品，调入仓库排除调出仓库。
- 退货原单会带出仓库、往来单位、价格和剩余数量。
- 基础资料详情和关联原单详情入口存在。
- 执行完整 Vitest 测试和生产构建。

默认 Maven 测试不依赖 Docker；真实 MySQL 场景使用 `mysql-integration` profile 验证。

## 验收标准

- 用户不再需要输入仓库 ID、商品 ID、客户 ID、供应商 ID 或退货原单号。
- 新增和修改不能提交不存在、已停用或跨工作区的关联资料。
- 退货商品和数量始终受原单及剩余可退数量限制。
- 基础资料可以查看完整详情，退货详情可以继续查看来源原单。
- 现有采购、销售、调拨、审核、库存和结算流程不回归。
